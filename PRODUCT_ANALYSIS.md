# PRODUCT_ANALYSIS.md — VisionWorkbench

## 产品定位

VisionWorkbench 是一款**本地优先的轻量级 AI 视觉研发工作台**，面向学生、创客、竞赛团队和小型研究组，覆盖从数据集到演示的完整计算机视觉工作流。

---

## 1. 目标用户画像

### 主要用户

| 用户群体 | 典型场景 | 核心痛点 |
|----------|----------|----------|
| 本科生/研究生 | 课程项目、毕业设计 | 工具碎片化，流程不连贯 |
| 竞赛团队 | RoboMaster、无人机、智能车 | 实验管理混乱，模型版本丢失 |
| 创客/Maker | 树莓派、OpenMV、OpenART | 缺乏系统化工具，训练历史无处追溯 |
| YOLO 用户 | 目标检测、实例分割 | 仅有训练命令，缺乏前后环节支持 |
| 本科生研究员 | 小规模 CV 研究 | 实验不可复现，结果难以整理 |

### 次要用户

| 用户群体 | 典型场景 | 核心痛点 |
|----------|----------|----------|
| 小型实验室 | 2-5 人协作项目 | 数据和模型组织无序 |
| 初创原型团队 | 快速验证 CV 想法 | 工具链搭建成本高 |

### 非目标用户

- 企业级 MLOps 团队（需要 Kubernetes、分布式系统）
- 大规模云平台用户
- DevOps 重型团队

## 2. 问题空间分析

### 2.1 当前工作流的碎片化

典型的 CV 学生项目使用 5-6 个独立工具：

```
LabelImg/CVAT → YOLO CLI → Excel → 本地文件夹 → PPT 截图
   (标注)       (训练)    (实验记录)  (模型存储)   (报告)
```

这导致 6 个核心问题：

| 问题 | 影响 | 严重度 |
|------|------|--------|
| 数据集组织混乱 | 找不到正确的数据集版本 | 高 |
| 实验不可复现 | 无法重现之前的训练结果 | 高 |
| 模型版本丢失 | 最佳模型找不回来 | 高 |
| 训练历史消失 | 不知道之前试过什么参数 | 中 |
| 失败分析困难 | 不知道为什么效果不好 | 中 |
| 协作混乱 | 团队成员信息不同步 | 中 |

### 2.2 用户旅程地图

```
现状 (As-Is):

标注图片 ──→ 手动整理文件夹 ──→ 写训练命令 ──→ 截图记录结果 ──→ 手动对比
  ↓              ↓                ↓              ↓              ↓
用LabelImg    命名混乱        参数丢失        Excel散落       PPT拼凑

理想 (To-Be, VisionWorkbench):

注册数据集 ──→ 验证数据集 ──→ 启动训练 ──→ 记录实验 ──→ 对比评估 ──→ 演示
    ↓             ↓            ↓           ↓           ↓          ↓
 结构化存储    自动校验    参数追踪   自动对比   指标可视化  一键演示
```

## 3. 价值主张

### 3.1 核心差异化

| 维度 | 现状 | VisionWorkbench |
|------|------|-----------------|
| 部署方式 | 云服务或碎片化工具 | **本地一键运行** |
| 工作流覆盖 | 单一环节 | **6 个核心模块全覆盖** |
| 学习曲线 | 陡峭（需学习多个工具） | **大一学生可理解** |
| AI 集成 | 无或事后添加 | **AI Agent 是头等公民** |
| 数据隐私 | 上传到云服务 | **数据完全本地** |
| 可复现性 | 手动记录，经常遗漏 | **自动追踪全链路** |

### 3.2 一句话价值主张

> 不要让 CV 工作流被碎片化工具绑架——用 VisionWorkbench，一个本地工具管理从数据到演示的完整流程。

### 3.3 成功标准

用户使用 VisionWorkbench 后能够清晰回答：

1. 哪个数据集训练了这个模型？
2. 训练用了什么参数？
3. 哪个实验提升了性能？
4. 应该部署哪个模型？
5. 模型还在犯什么错误？
6. 如何快速展示结果？

## 4. 产品原则

### 原则 1：本地优先
- 系统必须在个人电脑上完整运行
- 互联网连接应为可选项
- 数据不上传、不依赖云服务

### 原则 2：学生友好
- 大一学生应当能理解工作流
- 避免企业级复杂性
- 避免 DevOps 重型架构
- 界面直观，交互简洁

### 原则 3：AI 原生
- Claude、Codex、Cursor 等 AI 工具为头等公民
- 系统设计假设 AI 辅助存在
- 预留 AI Agent 接口层
- 文件结构和 API 设计考虑 AI 可读性

### 原则 4：工程优先于论文
- 目标不是发论文，是提升生产力
- 关注：可复现性、组织性、实验性、评估性、报告性

### 原则 5：渐进式开发
- 每个版本必须可用
- 不在验证价值前构建庞大架构
- 从单模型推理起步，逐步扩展

## 5. 市场定位

### 5.1 竞争格局坐标

```
                  云端
                   |
        Roboflow   |   Ultralytics Platform
        (标注+训练) |   (全流程)
                   |
    ───────────────┼──────────────────
        单环节     |      全流程
                   |
    CVAT           |   VisionWorkbench ★
    LabelImg       |   Fovux
    Label Studio   |
                   |
                  本地
```

### 5.2 市场空白

- **左上**（云+单环节）：CVAT/Label Studio — 强标注，无训练/实验/模型管理
- **右上**（云+全流程）：Ultralytics Platform — 全流程但云绑定，需要网络
- **左下**（本地+单环节）：LabelImg — 本地标注，功能极简
- **右下**（本地+全流程）：**市场空白** — VisionWorkbench 的目标位置

Fovux 在本地+全流程象限，但它是 MCP 协议驱动的 CLI/VS Code 工具，不是独立 GUI 工作台，面向的是已熟悉 MCP 的开发者。

### 5.3 目标市场规模（估算）

- 全球 CV 相关本科/研究生：每年 50-100 万
- 竞赛团队（RoboMaster 等）：每年 3000-5000 支队伍
- OpenMV/OpenART 用户：10 万+
- Maker 社区 CV 项目：5 万+
- 小型实验室和研究组：1-2 万

## 6. 模块分析

### 6.1 核心模块

| 模块 | 职责 | 回答的关键问题 |
|------|------|---------------|
| **Dataset Center** | 数据集注册、元数据、统计、校验、版本 | 我有哪些数据？标注对了吗？ |
| **Training Center** | 启动训练、记录参数、管理任务、日志 | 什么在训练？为什么失败了？ |
| **Experiment Center** | 实验记录、比较、笔记、结论 | 改了什么？有改进吗？谁最好？ |
| **Model Registry** | 模型版本、元数据、比较、血统 | 模型从哪来？该部署哪个？ |
| **Evaluation Center** | 图片/视频/批量评估、指标、错误分析 | 模型哪里犯错？怎么改进？ |
| **Demo Center** | 图片/视频/摄像头演示 | 怎么快速展示？ |

### 6.2 未来 AI Agent 层

| Agent | 能力 |
|-------|------|
| Dataset Agent | 检测标注问题、分析数据分布 |
| Training Agent | 分析训练失败、建议超参数 |
| Experiment Agent | 对比实验、生成结论 |
| Evaluation Agent | 错误聚类、改进建议 |
| Documentation Agent | 生成技术文档 |
| Presentation Agent | 生成 PPT 大纲、图表 |

## 7. 技术约束

### 推荐技术栈

| 类别 | 选型 |
|------|------|
| 语言 | Python 3.10+ |
| 训练框架 | Ultralytics YOLO、PyTorch |
| 视觉库 | OpenCV |
| UI 框架 | Gradio / Streamlit |
| 数据库 | SQLite |
| 包管理 | uv / pip |

### 明确避免

- Kubernetes
- 微服务架构
- 分布式系统
- 纯云架构
- 企业 DevOps 复杂性

## 9. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| Ultralytics 强化本地功能 | 中 | 高 | 差异化：AI 原生+全模块覆盖 |
| Gradio/Streamlit 限制复杂 UI | 中 | 中 | 从 Streamlit 起步，预留自定义前端路径 |
| 用户习惯现有碎片化工具 | 高 | 中 | 极致简单的一键体验，降低迁移成本 |
| Python 依赖管理问题 | 高 | 低 | uv 锁文件 + 虚拟环境封装 |
| 跨平台兼容性 | 中 | 中 | 从 Windows 起步，逐步扩展 |

## 10. 总结

VisionWorkbench 的核心价值在于**统一**——将 5-6 个碎片化工具合并为一个有凝聚力的工作台，同时保持本地运行、简单易用。它不试图替代任何一个巨头产品，而是填补一个被忽视的市场空白：**学生和创客需要的轻量级、本地优先、全流程 CV 工作台**。

竞争优势不在于功能的绝对数量，而在于：
- **工作流连贯性**：一个工具完成所有事情
- **零配置体验**：安装即用，无需 Docker/K8s
- **AI 原生设计**：为 AI 辅助工作流设计
- **学习友好**：降低 CV 研发的入门门槛
